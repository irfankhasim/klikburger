PART 1 — SYSTEM OVERVIEW
================================================================================

Project purpose
---------------
Klik Burger — POS (Point of Sale) and back-office system for a burger outlet:
counter sales, receipts, order/kitchen lists, cash drawer/shifts, ingredient
costing (FIFO), menu costing, staff, and monthly reports. Backend data is
Firestore + Firebase Auth. No custom REST API server found in repo.

Main features (from code)
---------------------------
- Authentication: email/password Firebase (html/login.html + js/login.js).
- Main menu shell: POS vs Back Office navigation, iframe embeds, route restore
  (html/main-menu.html, js/main-menu.js).
- Client RBAC: OWNER, ADMIN, SHIFT_LEAD, CASHIER; clock in/out; drawer/shift
  gates; read-only mode after shift close (js/pos-rbac-session.js).
- POS sales: cart, batch stock checks, Firestore checkout with FIFO deduction,
  sales + POS order/receipt docs (js/pos-order-app.js, js/pos-sale-fifo.js,
  js/pos-checkout-firestore-writer.js).
- POS operations hub: real-time orders, receipts, shift, cash movements, audit
  (js/pos-firestore-hub.js, re-exported via js/pos-operations-hub.js).
- Costing & inventory: ingredients, ingredient_batches, ingredient_ledger,
  modifiers (legacy), recipes, menu_items, purchase_history — schema notes in
  js/firebase/collections.js.
- Staff: CRUD staff, activity, KPI settings (js/staff/staff-repository.js and
  staff/js/ mirrors).
- Monthly reports: aggregate to monthly_reports/{YYYY-MM}
  (js/monthly-reports/generate-monthly-report.js, html/bo-monthly-reports.html).
- Landing: html/landing.html; index.html redirects to landing.
- Dev scripts: seed Firestore/Auth, deploy rules/indexes, emulators
  (scripts/*.mjs, package.json npm scripts).
- MCP (existing): stdio server, read-only Firestore via Admin SDK
  (mcp-server.mjs, .cursor/mcp.json).

User roles
----------
- Firestore users/{uid} (or lookup by email): role mapped to RBAC
  (js/pos-firebase-auth-bridge.js; seed example: scripts/seed-demo-auth-users.mjs).
- Client RBAC (js/pos-rbac-session.js): ROLES.OWNER, ADMIN, SHIFT_LEAD, CASHIER;
  back office only OWNER/ADMIN (canAccessBackOfficeModule).

Main workflows
--------------
1. User -> login.html -> Firebase Auth -> getPosUserRbacPayload -> loginSession
   -> main-menu.html.
2. Staff POS: clock in -> (if needed) open shift drawer -> sale/checkout ->
   receipts; close shift -> read-only per RBAC.
3. Owner/Admin: BO modules (dashboard, staff, cost calculator, monthly reports)
   via shell menu.
4. Checkout: finalizePosSaleFifo — transaction: read/update ingredient_batches,
   write sales, update pos_meta/counters, write pos_orders (+ subcollection items),
   pos_receipts, pos_sales_transactions.

System architecture summary
-----------------------------
| Layer   | Technology | Evidence |
|---------|------------|----------|
| UI      | Static HTML + CSS + ES modules (Firebase client from CDN) | js/firebase/init.js |
| Data    | Cloud Firestore (or emulator) | firebase.json, js/firebase/init.js |
| Auth    | Firebase Auth (email/password) | js/login.js, js/firebase/init.js |
| Hosting | Firebase Hosting, public: "." | firebase.json |
| Admin/AI| Node + firebase-admin, seed scripts, mcp-server.mjs | package.json, scripts/lib/admin-init.mjs |

No Cloud Functions folder; no getStorage / firebase-storage; no getDatabase for
Realtime DB in code (databaseURL exists in js/firebase/config.js but unused).

PART 5 — BUSINESS LOGIC
================================================================================

User flow
---------
Login -> RBAC payload -> main menu -> (POS) clock in per role -> embed sales/receipts/order board.

Admin flow
----------
OWNER/ADMIN can switch to Back Office (canAccessBackOfficeModule in
js/pos-rbac-session.js); BO nav in js/main-menu.js / html/main-menu.html.

Approval flow
-------------
No explicit approval workflow in code; financial void uses prototype manager PIN
(js/pos-security-constants.js, voidReceiptInHub in js/pos-firestore-hub.js).

Automation flow
---------------
- Monthly: generateAndWriteMonthlyReport — timestamp range aggregation
  (js/monthly-reports/generate-monthly-report.js).
- Seed: scripts/setup.mjs -> runAllSeeds.

Data flow
---------
Read: onSnapshot / getDocs.
Sale write: single large runTransaction in finalizePosSaleFifo — batch updates,
create sales, appendCheckoutInTransaction for POS docs
(js/pos-sale-fifo.js + js/pos-checkout-firestore-writer.js).
Shift: runTransaction open shift, update pos_meta/counters.activeShiftDocId
(js/pos-firestore-hub.js).

CRUD flow
---------
Repository pattern (addDoc, updateDoc, deleteDoc, setDoc) for staff, ingredients,
etc.; POS uses transactions/batches as above.

PART 6 — MCP PREPARATION
================================================================================

Current state
-------------
- mcp-server.mjs: MCP @modelcontextprotocol/sdk (stdio), firebase-admin via
  getAdminFirestore(); READ-ONLY tools:
  get_menu_items, get_orders, get_staff_list, get_sales_summary, get_ingredients.
- .cursor/mcp.json: node mcp-server.mjs from workspace root.
- Credentials: same as scripts — GOOGLE_APPLICATION_CREDENTIALS,
  firebase-service-account.json, or emulator env (scripts/lib/admin-init.mjs).

Suggested production MCP architecture
-------------------------------------
1. Layering: MCP tools -> thin Node module -> Firestore Admin; no raw wide write.
2. Tools by domain: orders, inventory, staff, reports — align with
   js/firebase/collections.js.
3. Separate read vs write tools; writes whitelisted (e.g. append_audit_log,
   regenerate_monthly_report) with input validation.
4. External MCP auth: do not expose service account to agent clients.

Security model & permission layer
---------------------------------
Current Firestore rules are open (if true) — Admin SDK bypasses rules; control
must be at MCP server + IAM (least-privilege SA, or client read-only tools only).
Role gate: only trusted humans connect write-capable MCP.

Recommended MCP structure
-------------------------
tools/read/   — bounded queries, pagination
tools/write/ — small whitelisted actions + idempotency
resources/    — project id, schema version snapshot

AUTH vs STAFF (MODEL OPERASI)
================================================================================

Firebase Auth — hanya dua jenis akaun utama untuk sistem ini:
  - Owner (contoh: Irfan) — akses BO + POS mengikut RBAC.
  - Staff kongsi (contoh: Ikhwan) — satu akaun Auth untuk semua pekerja kaunter.

Koleksi Firestore `staff` — BUKAN akaun log masuk individu. Ia menyimpan data
operasi & pemantauan: peranan kerja, jadual, aktiviti, analytics, dan ID yang
dicatat pada jualan apabila pekerja memilih nama mereka di menu utama (medan
sesi `operationalStaffId` / `operationalStaffName` dalam pos-rbac-session.js).

MCP / Admin SDK: urus dokumen `staff` & `staff_activity` tanpa mencipta
pengguna Auth baru untuk setiap pekerja. Auth demo kekal pada
scripts/seed-demo-auth-users.mjs (dua pengguna).
